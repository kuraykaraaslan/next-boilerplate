# PLAN/8 — Tenant-bazlı Aç/Kapa + Plugins Ekranı

**Bağımlılık:** PLAN/6. **Paralellik:** PLAN/7, PLAN/9 ile eşzamanlı.
**Önce oku:** `PLAN/0.md`, `PLAN/6.md`.

## Amaç
Modülleri WordPress "Plugins" ekranı gibi tenant başına runtime aç/kapa yap. Durum `setting` store'da; ileride app-store kurulumu aynı anahtarları yazabilir.

## Yapılacaklar
1. **`@nb/setting/server/module-activation.service.next.ts`**: `getEnabledModuleIds(tenantId): Promise<Set<string>>`.
   - Anahtar: `module.<id>.enabled`. `SettingService.getByKeys(tenantId, keys)` (✓ mevcut, batched) ile toplu oku.
   - Anahtar yoksa → manifest `enabled` (default true); varsa `=== 'true'`.
2. **`app/tenant/[tenantId]/api/modules/route.ts`** (yeni) — `admin-settings/route.ts` şablonu (auth/rate-limit/error aynen):
   - `GET` → `{ modules: [{ id, name, icon, enabled }] }` (runtime.modules + getEnabledModuleIds merge), admin-gated.
   - `PUT { id, enabled }` → `SettingService.updateMany(tenantId, { [key]: String(enabled) })` (✓ mevcut). İleride dependency kuralı (requires eden varsa kapatma engeli) burada.
3. **`@nb/setting/ui/ModuleManagerPage.tsx`** + **`app/tenant/[tenantId]/admin/modules/page.tsx`**: toggle'lı liste; `PUT /api/modules`'a vurur. Bu sayfanın kendisi `setting` manifestinde bir menu item.
4. **`ModuleEnabledContext` beslemesi**: admin layout (server) `getEnabledModuleIds(tenantId)` çağırır, sonucu provider'a + AdminShell'e geçer (her navigasyonda client fetch yok).

## Bitti tanımı (DoD)
- `/tenant/<id>/admin/modules`'tan `blog` kapat → `module.blog.enabled=false` yazılır → reload → Blog menü grubu + slot katkıları kaybolur → aç → geri gelir.
- Tenant-bazlı izole (başka tenant etkilenmez).

## Dokunulan dosyalar
`@nb/setting/server/module-activation.service.next.ts` (yeni), `app/tenant/[tenantId]/api/modules/route.ts` (yeni), `@nb/setting/ui/ModuleManagerPage.tsx` + `app/tenant/[tenantId]/admin/modules/page.tsx` (yeni), admin layout (provider besleme).
